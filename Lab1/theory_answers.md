# ЛР1 — Ответы на контрольные вопросы

---

## 1. Разновидности и назначение систем сборки

### Разновидности (минимум 3):

1. **Make-подобные** — GNU Make, NMake, MinGW Make. Работают на основе файлов `Makefile`, описывающих правила и зависимости между целями.
2. **Мета-системы сборки (генераторы)** — CMake, qmake, Meson. Генерируют файлы для нижележащих систем сборки (Makefile, Ninja, Visual Studio projects) из своего формата описания проекта.
3. **Встроенные в IDE** — MSBuild (Visual Studio), Xcode Build System. Тесно интегрированы в конкретную среду разработки, используют проектные файлы IDE (.vcxproj, .xcodeproj).
4. **Скриптовые** — Ninja, SCons, Gradle (для C++ через плагины). Ninja — минималистичная быстрая система; SCons использует Python-скрипты; Gradle — на Groovy/Kotlin.

### Назначение (функции) системы сборки (минимум 3):

1. **Компиляция исходного кода** — вызов компилятора для преобразования .cpp → .o / .obj файлов с нужными флагами и путями включения.
2. **Отслеживание зависимостей** — определение, какие файлы изменились, и перекомпиляция только того, что затронуто изменениями (инкрементальная сборка).
3. **Линковка** — объединение объектных файлов и библиотек в исполняемый модуль (.exe, .dll, .so).
4. **Управление конфигурациями** — поддержка профилей Debug/Release с разными флагами оптимизации, отладочной информацией и т.п.
5. **Кроссплатформенность** — абстракция от конкретного компилятора и ОС (особенно у мета-систем типа CMake).

## 2. Преимущества и недостатки CMake

### Преимущества:

1. **Кроссплатформенность** — один и тот же CMakeLists.txt работает под Windows (MSVC, MinGW), Linux (GCC, Clang), macOS (Clang). CMake генерирует нативные файлы сборки для каждой платформы.
2. **Широкая поддержка экосистемы** — поддержка большинства IDE (CLion, Visual Studio, Qt Creator), пакетных менеджеров (vcpkg, Conan), CI-систем. Стал де-факто стандартом для C++ проектов.

### Недостатки:

1. **Сложный и неинтуитивный синтаксис** — язык CMake исторически не имеет строгой типизации, операции со строками и списками неочевидны, отладка скриптов затруднена.
2. **Медленная генерация на больших проектах** — фаза конфигурирования (cmake ..) может занимать значительное время на крупных проектах с сотнями модулей, особенно при поиске зависимостей (find_package).

## 3. static_cast<> и reinterpret_cast<>

### Отличия:

| Характеристика | `static_cast<>` | `reinterpret_cast<>` |
|---|---|---|
| **Проверка типов** | Компилятор проверяет допустимость преобразования на этапе компиляции | Никаких проверок — прямая переинтерпретация битового представления |
| **Допустимые преобразования** | Числовые типы (int→float, int→enum), указатели вверх/вниз по иерархии наследования, void*→T* | Любые указатели друг в друга, указатели ↔ целые числа |
| **Безопасность** | Относительно безопасен, UB только при неверном downcast без виртуальных функций | Опасен — легко нарушить strict aliasing, получить UB |
| **Семантика** | Логическое преобразование значения (например, округление float→int) | Переинтерпретация битов без изменения |

### Машинные инструкции:

- **`static_cast<int>(floatVal)`** — генерирует инструкцию преобразования, например `cvttss2si` (SSE: convert float to int with truncation) или `cvttsd2si` (для double). Реально меняет битовое представление.
- **`static_cast<Base*>(derivedPtr)`** — при одиночном наследовании: **никаких инструкций** (адрес тот же). При множественном наследовании: `add`/`sub` для коррекции смещения подобъекта.
- **`reinterpret_cast<unsigned char*>(ptr)`** — **никаких инструкций**. Это чисто компиляторная директива: тот же адрес интерпретируется как указатель на другой тип. На уровне машинного кода — `nop` (или ничего).

### Пример из ЛР1 (`cryptoutils.cpp`, строка 31):
```cpp
reinterpret_cast<const unsigned char*>(salt.constData())
```
Здесь `const char*` переинтерпретируется как `const unsigned char*` для OpenSSL API. Машинная инструкция: отсутствует (тот же адрес в регистре).

## 4. Передача по ссылке vs по значению

### Передача по ссылке — преимущества:

**Производительность:**
- Не происходит копирование объекта — передаётся только адрес (8 байт на x64). Для крупных объектов (QString, QByteArray, std::vector) это критически важно — копирование может включать выделение памяти в куче и побайтовое копирование данных.
- Нет вызова конструктора копирования и деструктора для параметра.

**Безопасность (const-ссылка):**
- `const &` гарантирует на уровне компилятора, что функция не модифицирует переданный объект.
- Нельзя случайно изменить оригинал — компилятор выдаст ошибку.

### Сравнение:

| Критерий | По значению | По ссылке (`const T&`) |
|---|---|---|
| Копирование | Да (полная копия) | Нет (передаётся адрес) |
| Стоимость для `QString` | Аллокация + memcpy | 8 байт (указатель) |
| Защита от модификации | Изменяется копия, оригинал цел | `const` запрещает изменение |
| Для примитивов (int, double) | Эффективно (помещается в регистр) | Избыточно (косвенная адресация медленнее) |
| Время жизни | Копия живёт в scope функции | Ссылка валидна пока жив оригинал |

### Рекомендация:
- Примитивные типы (int, double, bool, указатели) — **по значению**
- Составные типы (QString, QByteArray, std::string, структуры) — **по `const &`**

## 5. Угрозы на защищённое локальное хранилище паролей

### 6 видов атак и методы защиты:

#### 1. Кража файла учётных данных из файловой системы
- **Атака:** злоумышленник копирует файл хранилища с диска (физический доступ, вредоносное ПО, резервные копии).
- **Защита:** (a) шифрование файла AES-256 с ключом, производным от мастер-пароля (PBKDF2); (b) ограничение прав доступа к файлу (ACL).
- **Нарушитель:** базовые возможности (уровень 1 по Приложению 8 ФСТЭК) — имеет доступ к ФС через пользовательский интерфейс ОС.

#### 2. Снятие дампа оперативной памяти (Credential Dumping)
- **Атака:** злоумышленник создаёт дамп процесса менеджера паролей и извлекает расшифрованные пароли из памяти.
- **Защита:** (a) двухслойное шифрование — пароли расшифровываются поштучно по запросу, а не все сразу; (b) немедленная очистка расшифрованных данных из памяти (SecureZeroMemory).
- **Нарушитель:** базовые повышенные возможности (уровень 2) — может запускать произвольные инструменты, в т.ч. утилиты создания дампов (ProcDump, Task Manager).

#### 3. Отладка на пользовательском уровне
- **Атака:** злоумышленник подключает отладчик (x64dbg, WinDbg) к процессу менеджера паролей, ставит breakpoint на функцию расшифровки и считывает данные.
- **Защита:** (a) IsDebuggerPresent() при запуске; (b) самоотладка — приложение-спутник занимает единственный слот отладчика через DebugActiveProcess().
- **Нарушитель:** базовые повышенные возможности (уровень 2) — имеет пользовательские привилегии и может запускать отладчики.

#### 4. Модификация (патч) исполняемого файла
- **Атака:** злоумышленник изменяет байты в .exe для обхода проверок (NOP-out проверки пин-кода, отключение антиотладки).
- **Защита:** (a) самопроверка контрольной суммы (SHA-256 хеш сегмента .text при запуске); (b) цифровая подпись исполняемого файла (Authenticode).
- **Нарушитель:** базовые повышенные возможности (уровень 2) — имеет доступ к ФС и hex-редакторам.

#### 5. Перехват ввода (Keylogging)
- **Атака:** кейлоггер перехватывает нажатия клавиш при вводе мастер-пароля.
- **Защита:** (a) использование защищённого поля ввода (виртуальная экранная клавиатура); (b) аппаратные ключи (FIDO2/WebAuthn) вместо PIN-кода.
- **Нарушитель:** базовые возможности (уровень 1) — может установить вредоносное ПО с функцией перехвата ввода.

#### 6. Атака по побочным каналам (Side-channel)
- **Атака:** анализ времени выполнения операций шифрования, энергопотребления или электромагнитного излучения для восстановления ключа.
- **Защита:** (a) использование криптографических библиотек с constant-time реализацией (OpenSSL); (b) PBKDF2 с большим количеством итераций (100 000), что делает timing-атаки менее информативными.
- **Нарушитель:** средние возможности (уровень 3) — обладает знаниями о криптографических реализациях и специализированным оборудованием.

## 8. Сегменты в PE-файле

### Что такое сегменты:

Сегменты (секции) в PE-файле (Portable Executable) — это логически разделённые области, каждая из которых содержит данные определённого типа. При загрузке PE-файла в виртуальную память загрузчик ОС размещает каждый сегмент по своему виртуальному адресу с определёнными правами доступа (чтение, запись, исполнение).

### Основные сегменты:

| Сегмент | Содержимое | Права |
|---------|-----------|-------|
| **.text** | Машинный код (скомпилированные инструкции) | Read + Execute |
| **.rdata** | Константы, строковые литералы, таблицы импорта/экспорта, RTTI | Read |
| **.data** | Инициализированные глобальные и статические переменные | Read + Write |
| **.bss** | Неинициализированные глобальные и статические переменные | Read + Write |
| **.rsrc** | Ресурсы (иконки, диалоги, манифест, версия) | Read |
| **.reloc** | Таблица релокаций (для ASLR) | Read |

### Куда помещается машинный код:

В сегмент **.text**. Именно его адрес и размер мы получаем в `integritycheck.cpp` через разбор PE-заголовков.

### Влияние изменений в исходном коде:

- **Изменение состава/порядка операторов** (логика, условия, циклы, вызовы функций) → изменяется сегмент **.text** (другие машинные инструкции).
- **Изменение строковых констант** (`"Hello"` → `"World"`) → изменяется сегмент **.rdata** (секция с read-only данными). Сегмент .text **не изменяется**, если инструкции остались теми же (те же вызовы, те же ветвления).
- **Изменение значений инициализированных глобальных переменных** → изменяется сегмент **.data**.
- **Изменение числовых констант, встроенных в инструкции** (immediate operands, например `mov eax, 42` → `mov eax, 99`) → изменяется сегмент **.text**, т.к. константа закодирована непосредственно в машинной инструкции.

## 9. Виртуальная память

### Определение:

Виртуальная память — это механизм управления памятью, реализуемый ОС совместно с аппаратным обеспечением (MMU — Memory Management Unit), который предоставляет каждому процессу собственное изолированное непрерывное адресное пространство, абстрагированное от физической оперативной памяти.

### Назначение:
- Изоляция процессов друг от друга (каждый процесс «видит» только свою память).
- Предоставление программам большего объёма памяти, чем физически доступно (за счёт подкачки на диск).
- Упрощение программирования — программа работает с непрерывным адресным пространством.

### Преимущества:
- **Изоляция и безопасность** — процесс не может прочитать/записать память другого процесса.
- **Гибкость** — суммарный объём виртуальной памяти всех процессов может превышать объём физической RAM.
- **Упрощение загрузки** — каждый процесс может загружаться по одному и тому же виртуальному адресу (с учётом ASLR — по случайному, но валидному).

### Недостатки:
- **Накладные расходы** — трансляция адресов через таблицу страниц требует дополнительных обращений к памяти (смягчается TLB-кэшем).
- **Page faults** — при подкачке с диска (swap) возникают значительные задержки (миллисекунды vs наносекунды для RAM).
- **Фрагментация** — внутренняя фрагментация из-за выравнивания по страницам (обычно 4 КБ).

### Соотношение виртуальных и физических адресов:

Виртуальный адрес транслируется в физический через **таблицу страниц** (page table), управляемую ОС:

1. Виртуальный адрес разбивается на **номер страницы** (старшие биты) и **смещение внутри страницы** (младшие 12 бит для страниц 4 КБ).
2. Номер виртуальной страницы используется как индекс в таблице страниц процесса.
3. Таблица страниц содержит номер соответствующей физической страницы (page frame).
4. Физический адрес = номер физической страницы + смещение.

Один виртуальный адрес в разных процессах может указывать на разные физические страницы. Разные виртуальные адреса могут указывать на одну физическую страницу (shared memory, memory-mapped files).

## 10. Виды библиотек по типу подключения

### 3 вида:

#### 1. Статические библиотеки (.lib / .a)

- **Особенность:** код библиотеки встраивается (линкуется) непосредственно в исполняемый файл на этапе сборки.
- **Преимущества:** (a) нет внешних зависимостей при запуске — всё в одном .exe; (b) чуть выше производительность — нет накладных расходов на динамическую загрузку и трансляцию адресов.
- **Недостатки:** (a) увеличивает размер исполняемого файла; (b) при обновлении библиотеки нужно пересобирать все приложения, использующие её.

#### 2. Динамические библиотеки (.dll / .so)

- **Особенность:** код библиотеки находится в отдельном файле, загружаемом ОС при запуске приложения (implicit linking — через import library .lib/.dll.a).
- **Преимущества:** (a) несколько приложений могут разделять одну копию библиотеки в памяти; (b) обновление библиотеки не требует перекомпиляции приложения.
- **Недостатки:** (a) «DLL Hell» — проблема совместимости версий; (b) DLL должна быть доступна при запуске приложения, иначе — ошибка.

#### 3. Библиотеки с загрузкой во время выполнения (runtime / plug-in)

- **Особенность:** загружаются программно через API (`LoadLibrary`/`dlopen`), функции ищутся по имени (`GetProcAddress`/`dlsym`).
- **Преимущества:** (a) приложение может запускаться даже если библиотека отсутствует (graceful degradation); (b) позволяет реализовать плагинную архитектуру — подгружать модули по мере необходимости.
- **Недостатки:** (a) более сложный код — ручное управление загрузкой и получением адресов функций; (b) нет проверки типов на этапе компиляции — ошибки обнаруживаются только при выполнении.

### Библиотеки в ЛР1:

| Библиотека | Тип подключения | Файлы |
|------------|----------------|-------|
| **Qt 6** (Core, Gui, Widgets) | Динамическая (.dll) — Qt6Core.dll, Qt6Widgets.dll и т.д. копируются windeployqt в каталог с exe | Подключены через .pro: `QT += core gui widgets` |
| **OpenSSL** (libcrypto) | Динамическая (.dll) — libcrypto-3-x64.dll копируется в каталог с exe | Подключена через .pro: `LIBS += .../libcrypto-3-x64.dll` |
| **advapi32** | Системная динамическая (Windows API) — всегда присутствует в системе, линкуется через import library | Подключена через .pro: `LIBS += -ladvapi32` |

## 11. Защита в лабораторной работе — атаки и противодействие

### Реализованные защиты:

#### 1. Защита от кражи файла (AES-256 шифрование)

- **Как атакует:** копирует credentials.json.enc с диска → пытается прочитать содержимое.
- **С чем столкнётся:** файл зашифрован AES-256-CBC, ключ выводится из PIN через PBKDF2 (100 000 итераций). Без PIN невозможно расшифровать. Перебор PIN по файлу — крайне медленный из-за PBKDF2.
- **Как обойти:** brute-force PIN (если PIN короткий — 4 цифры = 10 000 вариантов, при 100 000 итераций PBKDF2 ≈ 0.1с на попытку = ~17 минут полный перебор).
- **Как предотвратить:** использовать длинный мастер-пароль вместо PIN; добавить задержку/блокировку после N попыток.

#### 2. Защита от дампа памяти (двухслойное шифрование)

- **Как атакует:** создаёт дамп процесса (Task Manager → Create dump file, или ProcDump) → ищет пароли в дампе.
- **С чем столкнётся:** в памяти хранятся только URL в открытом виде; логины и пароли зашифрованы вторым слоем и расшифровываются поштучно по запросу. После использования — очищаются SecureZeroMemory.
- **Как обойти:** снять дамп в момент, когда пользователь запросил расшифровку конкретного пароля (окно уязвимости очень короткое).
- **Как предотвратить:** минимизировать время жизни расшифрованных данных; использовать защищённые области памяти (VirtualLock, mlock).

#### 3. Защита от отладки (IsDebuggerPresent)

- **Как атакует:** подключает x64dbg к процессу → ставит breakpoint на CryptoUtils::decryptField → читает расшифрованные данные.
- **С чем столкнётся:** при запуске приложение вызывает IsDebuggerPresent() и блокирует интерфейс при обнаружении отладчика.
- **Как обойти:** пропатчить вызов IsDebuggerPresent (NOP), или использовать ScyllaHide (плагин для скрытия отладчика).
- **Как предотвратить:** множественные проверки в разных местах кода; обфускация; проверка целостности (пункт 5).

#### 4. Защита от отладки (самоотладка / Protector)

- **Как атакует:** пытается подключить отладчик к работающему менеджеру паролей.
- **С чем столкнётся:** слот отладчика уже занят Protector'ом — ОС отклоняет вторичное подключение (DebugActiveProcess вернёт ошибку).
- **Как обойти:** убить процесс Protector'а → подключить свой отладчик; или запустить PassManager напрямую без Protector.
- **Как предотвратить:** PassManager должен проверять наличие Protector'а; Protector не должен завершаться при попытке отсоединения (DebugSetProcessKillOnExit).

#### 5. Защита от модификации (хеш .text)

- **Как атакует:** открывает .exe в hex-редакторе → патчит проверку PIN (заменяет условный переход на безусловный) → запускает модифицированный exe.
- **С чем столкнётся:** при запуске вычисляется SHA-256 сегмента .text и сравнивается с эталоном — несовпадение → блокировка.
- **Как обойти:** (a) пропатчить саму проверку хеша; (b) пересчитать эталонный хеш после патча и записать его в .exe.
- **Как предотвратить:** вынести проверку в Protector (отдельный процесс); использовать несколько проверок в разных местах; обфускация кода проверки.

### Защита от 0-day уязвимостей:

0-day — уязвимость, о которой разработчик ещё не знает и для которой нет патча.

Методы защиты:
1. **Defence in depth** (эшелонированная защита) — множество уровней защиты, чтобы эксплуатация одной уязвимости не давала полный доступ.
2. **ASLR + DEP/NX** — рандомизация адресного пространства и запрет исполнения стека/кучи усложняют эксплуатацию memory corruption.
3. **Sandboxing** — запуск приложения в песочнице (AppContainer, seccomp) с минимальными привилегиями.
4. **Аномальное поведение** — системы обнаружения вторжений (IDS/IPS), мониторинг нетипичных вызовов API, SIEM.
5. **Минимизация поверхности атаки** — отключение неиспользуемых функций, минимальные привилегии процесса.
6. **Регулярное обновление** зависимостей (OpenSSL, Qt) для закрытия известных уязвимостей до их эксплуатации.

## 12. Стойкость симметричного шифрования и классификация

### Математическое обоснование стойкости:

Стойкость современных симметричных алгоритмов (AES) основана на принципе **вычислительной сложности**: не существует известного алгоритма, который позволяет восстановить ключ или открытый текст быстрее, чем **полный перебор ключей** (brute-force).

Для AES-256: пространство ключей = 2^256 ≈ 1.16 × 10^77. При скорости перебора 10^18 ключей/сек потребуется ≈ 10^59 лет — многократно превышает возраст Вселенной.

Конструкция AES (сеть замен-перестановок — Substitution-Permutation Network, SPN) математически обеспечивает:
- **Конфузию** (confusion) — каждый бит шифротекста зависит от нескольких битов ключа (через S-box).
- **Диффузию** (diffusion) — изменение одного бита открытого текста изменяет в среднем половину битов шифротекста (через ShiftRows и MixColumns).

Лучшие известные криптоатаки на AES (biclique) дают сложность 2^254.4 для AES-256 — практически не отличается от полного перебора.

### Классификация по длине шифруемых данных:

| Тип | Описание | Примеры |
|-----|----------|---------|
| **Блочные шифры** | Шифруют данные фиксированными блоками (64, 128, 256 бит). Для данных произвольной длины используются режимы (CBC, CTR, GCM и др.) + паддинг | AES (128 бит блок), DES (64 бит), ГОСТ 34.12-2018 «Кузнечик» (128 бит) |
| **Потоковые шифры** | Шифруют данные побитово/побайтово, генерируя псевдослучайный поток (keystream) из ключа | ChaCha20, RC4, Salsa20 |

### Классификация по числу ключей:

| Тип | Ключи | Описание | Примеры |
|-----|-------|----------|---------|
| **Симметричные** | 1 общий ключ | Один и тот же ключ для шифрования и расшифрования | AES, DES, ChaCha20, Blowfish |
| **Асимметричные** | 2 ключа (пара) | Открытый ключ для шифрования, закрытый — для расшифрования (или наоборот для подписи) | RSA, ECC, Диффи-Хеллман, ГОСТ Р 34.10-2012 |

## 13. Паддинг при шифровании

### Определение:

Паддинг (padding) — это добавление дополнительных байтов к открытому тексту перед шифрованием для того, чтобы его длина стала кратной размеру блока шифра. Необходим для блочных шифров в режимах, требующих полных блоков (CBC, ECB).

### Где используется в ЛР1:

Паддинг применяется неявно внутри OpenSSL при вызове `EVP_EncryptFinal_ex` и `EVP_DecryptFinal_ex`. По умолчанию OpenSSL EVP API использует PKCS#7 padding.

В `cryptoutils.cpp`, строка 111:
```cpp
if (EVP_EncryptFinal_ex(ctx,
                        reinterpret_cast<unsigned char*>(ciphertext.data()) + totalLen,
                        &outLen) != 1) {
```
Именно `EVP_EncryptFinal_ex` добавляет паддинг к последнему неполному блоку.

При расшифровке, строка 71:
```cpp
if (EVP_DecryptFinal_ex(ctx,
                        reinterpret_cast<unsigned char*>(plaintext.data()) + totalLen,
                        &outLen) != 1) {
```
`EVP_DecryptFinal_ex` проверяет и удаляет паддинг. Если паддинг некорректен (неверный ключ) — функция возвращает ошибку, что используется для определения неверного PIN-кода.

В Python-утилите `encrypt_credentials.py`, строка 46:
```python
padder = sym_padding.PKCS7(128).padder()
padded = padder.update(plaintext) + padder.finalize()
```
Здесь паддинг добавляется явно.

### Стандарты паддинга:

| Стандарт | Описание | Пример (блок 16 байт, данные 11 байт, нужно 5 байт паддинга) |
|----------|----------|------|
| **PKCS#7** (RFC 5652) | Каждый байт паддинга содержит число добавленных байтов. Если данные кратны блоку — добавляется полный блок паддинга | `... data [05 05 05 05 05]` |
| **ANSI X9.23** | Последний байт — число байт паддинга, остальные заполнены нулями | `... data [00 00 00 00 05]` |
| **ISO 10126** | Последний байт — число байт паддинга, остальные заполнены случайными байтами | `... data [random random random random 05]` |
| **ISO/IEC 7816-4** | Первый байт паддинга = 0x80, остальные — нули | `... data [80 00 00 00 00]` |
| **Zero padding** | Заполнение нулями. Проблема: невозможно отличить паддинг от данных, заканчивающихся нулями | `... data [00 00 00 00 00]` |
| **Нет паддинга (CTR, GCM)** | В потоковых режимах блочных шифров (CTR, OFB, CFB, GCM) паддинг не нужен — шифрование побитовое | — |
